I

The Data Pipeline

From raw shapefiles to cloud-native formats

Raw geospatial data is messy

What we get

  • Shapefiles (multi-file, projections vary)
  • GeoJSON (huge, no indexing)
  • GeoTIFF rasters (gigabytes each)
  • Scattered across FTP servers

What we need

  • Web-ready vector tiles
  • Query-ready columnar formats
  • Spatial indexing for fast joins
  • Unified metadata catalog

Spatial joins

Hive-partitioned on S3

H3: a universal hexagonal grid

  • Uber H3 discretizes Earth at 16 resolutions (res 0–15)
  • Res 8 ≈ 0.74 km² per cell — ~86M cells globally
  • Every point on Earth has a unique cell ID at every resolution
  • Equal-area hexagons — uniform neighbor distance, no polar distortion
  • Works identically for vectors (polygon → cells) and rasters (pixel → cell)

One grid. Any dataset. Any resolution.

H3 replaces polygon geometry with integer math

The problem: polygon operations

ST_Intersection, ST_Union, ST_Area —
O(n²), complex, error-prone on bad geometry

H3: hexagonize both, then set ops

Area = COUNT(DISTINCT cell_id) × cell_area
Replaces ST_* for area, join, intersection, buffer

All spatial ops become integer hash joins — O(n), no geometry library needed

Hive partitioning & why it matters

The layout

dataset/hex/
  h0=801fff.../data_0.parquet   ← North America
  h0=802fff.../data_0.parquet   ← Europe
  h0=803fff.../data_0.parquet   ← Asia
  ...

h0 = coarsest H3 cell = geographic region

Partition pruning

-- "Show me California data"
-- DuckDB reads ONLY the h0 partitions
-- that overlap California
-- → skips 90%+ of S3 reads

5–20× speedup on spatial queries

Four MCP tools

browse_stac_catalog
Discover available datasets
get_stac_details
S3 paths + column schemas
query(sql)
Execute DuckDB SQL against S3 parquet — H3 + httpfs extensions loaded
register_hex_tiles

-- Example: area of GAP 1+2 protected lands by state
SELECT state_name,
       APPROX_COUNT_DISTINCT(h8) * 0.7373 AS area_km2
FROM read_parquet('s3://public-biodiversity/padus/hex/h0=*/data_0.parquet')
WHERE gap_code IN (1, 2)
GROUP BY state_name ORDER BY area_km2 DESC

DuckDB on H3-indexed parquet

Per-request design

  • Fresh :memory: connection each request
  • Completely stateless — any pod can serve
  • Extensions: spatial, h3, httpfs
  • S3 credentials scoped per-request

Query rules (injected into tool description)

  • Always include h0 in joins
  • Use DISTINCT for vector overlaps
  • Use GROUP BY + agg for raster pixels
  • Never hardcode S3 paths — call get_stac_details first

The approval flow

I'll query the PAD-US dataset to rank states by the fraction of their area that is GAP status 1 or 2 (the strongest protection levels).
Details: query 
SELECT state_name,
  APPROX_COUNT_DISTINCT(h8) * 0.7373 AS protected_km2,
  total_km2,
  ROUND(protected_km2 / total_km2 * 100, 1) AS pct
FROM read_parquet('s3://...')
WHERE gap_code IN (1, 2)
GROUP BY state_name
ORDER BY pct DESC LIMIT 10

Design principles

STAC as single source of truth
Metadata, schemas, and asset paths all live in the catalog
Config over code
Zero JavaScript for app builders — just JSON + Markdown
MCP as universal interface
Same data server powers browsers, notebooks, CLIs, and agents
The virtuous training loop
Every deployment generates signal to improve metadata & prompts
H3 + hive partitioning
Fast queries at any scale — partition pruning skips continents
Separation of concerns
Data pipeline · MCP server · Agent frontend · LLM proxy — independent repos
